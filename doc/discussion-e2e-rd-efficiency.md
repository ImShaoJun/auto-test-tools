# 研发端到端提效体系设计讨论纪要

> 讨论时间：2026-05-22
> 参与方：产品/研发 + AI 助手

---

## 一、整体愿景

构建一套 AI 驱动的研发端到端提效体系，实现"需求一键交付"，打通从需求产出到代码交付再到质量验证的全链路。

## 二、全链路架构

```
┌─────────────┐     ┌──────────────────────────┐     ┌──────────────────────────┐
│   需求侧     │     │        研发侧              │     │        测试侧              │
│             │     │                          │     │                          │
│  Echo       │────▶│  Claude Code + MCP       │────▶│  Claude Code             │
│  ├─ 原始需求 │     │  ├─ 拉取上游产物（PRD/原型） │     │  ├─ 测试 Agent / MCP       │
│  ├─ 用户需求 │     │  ├─ 知识文件体系           │     │  ├─ 接口覆盖测试           │
│  ├─ PRD     │     │  ├─ 方案设计              │     │  ├─ 集成测试              │
│  └─ 原型    │     │  ├─ 代码实现              │     │  └─ 压力测试              │
│             │     │  └─ 单测用例              │     │                          │
└─────────────┘     └──────────────────────────┘     └──────────────────────────┘
```

### 各环节职责

| 环节 | 工具/平台 | 产物 | 交接方式 |
|------|----------|------|---------|
| **需求侧** | Echo | 原始需求、用户需求、PRD、原型 | MCP 暴露给研发侧拉取 |
| **研发侧** | Claude Code + MCP + 知识文件体系 | 方案设计、代码实现、单测用例 | 代码仓库 + CI/CD |
| **测试侧** | Claude Code + 测试 Agent / MCP | 接口覆盖、集成测试、压力测试 | 测试报告 |

## 三、知识文件体系（核心讨论）

研发侧的 AI 要精准高效地完成方案设计和代码实现，必须具备对业务和代码的深度理解。知识文件体系分为 **两种路线**：

### 3.1 第一种：Skill + CLAUDE.md + Repomix + catalog.md

#### 整体结构

```
代码仓库
│
├── CLAUDE.md                    ← 集成知识管理 Skill，约束 CC 行为
│
├── knowledge/                   ← 知识文件目录（由 Skill 管理）
│   ├── catalog.md               ← 知识文件的目录清单（索引）
│   ├── architecture.md          ← 架构设计文档
│   ├── business-concepts.md     ← 业务概念定义
│   ├── coding-standards.md      ← 编码规范
│   ├── module-xxx.md            ← 模块级业务知识
│   └── ...
│
└── repomix-output.xml           ← Repomix 生成的代码索引（单一文件）
```

#### 各组件职责

| 组件 | 职责 | 维护方式 |
|------|------|---------|
| **Skill** | 定义知识文件的组织规范、更新规则，约束 CC 按规范管理知识体系 | 人工设计，集成到 CLAUDE.md |
| **catalog.md** | 知识文件的目录清单，AI 通过它定位需要阅读的知识文件 | CC 在知识文件增删时自动更新 |
| **知识文件** | 业务知识、架构决策、编码规范等结构化文档 | 人 + CC 协作维护 |
| **Repomix 索引** | 代码的单一索引文件，AI 通过它快速理解代码结构、定位代码位置 | 自动生成（可挂 CI/git hooks） |

#### AI 的工作流

```
1. AI 接到需求（通过 MCP 从 Echo 拉取 PRD/原型）
2. AI 读取 catalog.md → 定位需要的知识文件 → 阅读业务知识
3. AI 在 Repomix 索引中检索 → 理解代码结构和现状
4. AI 结合业务知识 + 代码认知 → 精准完成方案设计和代码实现
```

#### 初始化过程

存量知识资产的梳理是最大的一次性成本：

1. **知识资产盘点** — 收集散落在 wiki、飞书、README、代码注释、PR 描述中的知识
2. **价值判断** — 判断哪些有效、哪些过时、哪些重复
3. **精简合并** — 人 + CC 协作，将有效知识整理成结构化文档
4. **纳入体系** — 按 Skill 规范组织到 knowledge/ 目录，更新 catalog.md
5. **生成代码索引** — 运行 Repomix 生成代码索引文件

### 3.2 第二种：DeepWiki / LLMWiki 知识图谱工具

#### 核心思路

使用自动化知识图谱工具（如 DeepWiki、LLMWiki）从代码仓库自动索引生成知识图谱，通过 API 提供查询能力。

#### 特点

| 维度 | 说明 |
|------|------|
| **知识来源** | 自动从代码仓库索引生成 |
| **知识组织** | 知识图谱 / 向量化索引 |
| **维护方式** | 工具自动重建 / 增量更新 |
| **使用方式** | 通过 API / MCP 查询 |
| **精准度** | 中（依赖自动理解质量，可能存在幻觉） |
| **维护成本** | 低（自动化） |

#### 主要挑战

- 自动生成的知识质量不可控，业务语义理解可能有偏差
- 私有部署有数据安全顾虑和算力成本
- 增量更新的频率和准确性需要权衡
- 与 CC 工作流的集成需要通过 MCP Server

### 3.3 两种方案对比

| 维度 | 第一种 (Skill + Repomix) | 第二种 (知识图谱工具) |
|------|------------------------|-------------------|
| **知识精准度** | ⭐⭐⭐⭐⭐ 人工审核把关 | ⭐⭐⭐ 自动生成，可能有幻觉 |
| **初始成本** | ⭐⭐⭐ 存量梳理成本高 | ⭐⭐⭐⭐ 工具开箱即用 |
| **持续维护成本** | ⭐⭐⭐ 依赖研发纪律 | ⭐⭐⭐⭐⭐ 自动化 |
| **外部依赖** | ⭐⭐⭐⭐⭐ 无需额外服务 | ⭐⭐ 需要部署服务/GPU |
| **数据安全** | ⭐⭐⭐⭐⭐ 全部本地化 | ⭐⭐⭐ SaaS 有安全顾虑 |
| **跨仓支持** | ⭐⭐⭐ 需额外设计 | ⭐⭐⭐⭐ 工具天然支持 |

## 四、第一种方案的成本与可行性评估

### 4.1 成本评估

#### 初始建设成本

| 项目 | 成本 | 说明 |
|------|------|------|
| Skill 开发 | 🟢 低 | 本质是 CLAUDE.md 里的规范 + prompt 约束 |
| catalog.md 设计 | 🟢 低 | 文件结构设计，工作量小 |
| Repomix 集成 | 🟢 低 | 成熟开源工具，`npx repomix` 即可 |
| **存量知识梳理** | 🔴 **高** | 每个仓库需要熟悉业务的研发投入 2-5 人天（中等复杂度），大型核心仓库可能 1-2 周 |

#### 持续维护成本

| 项目 | 成本 | 说明 |
|------|------|------|
| 知识文件更新 | 🟡 中 | Skill 约束 CC 同步更新，但执行力取决于研发纪律 |
| catalog.md 维护 | 🟢 低 | CC 可自动化 |
| Repomix 索引刷新 | 🟢 低 | 可挂 CI 或 git hooks 自动重新生成 |
| 知识质量审核 | 🟡 中 | 需要定期 review 知识文件是否过时 |

### 4.2 可行性评估

#### ✅ 可行的部分

1. **技术栈成熟** — CLAUDE.md、Skill、Repomix 都是现成的，不需要造轮子
2. **渐进式落地** — 可以先在 1-2 个核心仓库试点，不用一次铺开
3. **AI 使用体验好** — catalog.md + Repomix 索引给了 AI 清晰的导航地图
4. **知识精准度高** — 人工审核把关，不会出现知识图谱工具的幻觉问题
5. **无外部依赖** — 不需要额外部署服务、不需要 GPU、不需要上传代码到第三方

#### ⚠️ 风险与挑战

1. **知识腐化是最大的死因** — 体系建好后如果没人维护，3-6 个月就会变成过时的文档。需要制度保障（PR 模板强制检查、CI 检测等）
2. **跨仓库知识关联难做** — 单仓内 catalog.md 很清晰，多仓联合时知识如何引用 and 关联需要额外设计
3. **Repomix 索引的 token 上限** — 大型仓库即使 `--compress`，索引文件也可能很大，需要按模块拆分
4. **依赖研发主动参与** — 初始梳理需要熟悉业务的人投入，团队买账是前提

#### 综合评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术可行性 | ⭐⭐⭐⭐⭐ | 全部基于现有工具，技术上没有障碍 |
| 初始成本 | ⭐⭐⭐ | Skill/工具层低，存量知识梳理高 |
| 持续成本 | ⭐⭐⭐ | 取决于团队纪律和制度保障 |
| 效果上限 | ⭐⭐⭐⭐⭐ | 维护得好，AI 的精准度会非常高 |
| 风险 | ⭐⭐⭐ | 主要风险在人的因素，不是技术因素 |
